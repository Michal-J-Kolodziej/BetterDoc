PostgreSQL Connection Timeout on heavy write loads

warning

Symptom

Application throws 504 Gateway Timeouts or PG::ConnectionBad errors during peak traffic periods (usually 14:00 - 16:00 UTC). The error logs indicate that the connection pool is exhausted, and new worker threads hang indefinitely waiting for a database connection.

FATAL: remaining connection slots are reserved for non-replication superuser connections

Root Cause Analysis

The default connection pool size in our Rails database.yml is set to 5. However, during high concurrency events, our Sidekiq workers scale up to 25 threads per process. Since we deploy 4 pods, we are attempting to open 100 connections, while our RDS instance `max_connections` is capped at 80 for the current instance type.

PostgreSQL 14 Rails 7 Sidekiq
check_circle
build

Applied Fix

We need to implement PgBouncer for connection pooling to allow transaction-level pooling instead of session-level. Additionally, adjust the pool size in the configuration to better match available resources. 

# config/database.yml

production:
  adapter: postgresql
  encoding: unicode
  # Increase pool size to match sidekiq concurrency
  pool: 25
  host: <%= ENV['DATABASE_HOST'] %>
  username: <%= ENV['DATABASE_USER'] %>
  password: <%= ENV['DATABASE_PASSWORD'] %>
  # Enable prepared statements for PgBouncer compatibility
  prepared_statements: false

shield Prevention

  • Added Datadog monitor for postgresql.connections.active hitting 80% threshold.
  • Configured Sidekiq concurrency limit via ENV variables in Kubernetes deployment manifests.
  • Scheduled team training on Connection Pooling vs Session Pooling.
